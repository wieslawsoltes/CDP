# Page Domain

The `Page` domain in `CdpSession` manages the life cycle of the active application window, handles window navigation, reloads visual layouts, and provides high-DPI screenshot capturing and live screencasting functionality. It bridges the gap between browser-specific concepts (such as pages, frames, and load cycles) and Avalonia desktop window components.

The Page domain is defined and handled in `PageDomain.cs`. Automated test runners, inspector applications, and scripting agents can view and navigate the application state remotely using JSON-RPC over WebSockets.

---

## Overview of Supported Methods

The `PageDomain` maps browser navigation, layout, and visual streaming methods to Avalonia's window rendering and lifecycle. The following methods are implemented by `HandleAsync`:

| CDP Method | Description | Avalonia Backend Implementation |
|---|---|---|
| `Page.enable` / `Page.disable` | Enables or disables page event notifications. | Returns a success acknowledgment. |
| `Page.navigate` | Navigates the window to a specified URL. | Dynamically invokes `Navigate(url)` on the target window class. |
| `Page.reload` | Reloads the current page/window. | Forces a layout and visual invalidation cycle on the UI thread. |
| `Page.captureScreenshot` | Captures a high-DPI base64 PNG screenshot. | Renders the target window using `RenderTargetBitmap`. |
| `Page.startScreencast` | Starts streaming live window frames. | Spawns a background frame-capturing and encoding loop. |
| `Page.stopScreencast` | Stops live frame streaming. | Disposes of bitmap caches and unhooks rendering event listeners. |
| `Page.screencastFrameAck` | Acknowledges receipt of a screencast frame. | Releases the backpressure semaphore block in the streaming loop. |
| `Page.getLayoutMetrics` | Retrieves the viewport size dimensions. | Exposes the current client size dimensions of the window. |
| `Page.printToPDF` | Exports the window visual tree to a PDF document. | Renders the window to a SkiaSharp-backed PDF document. |
| `Page.getResourceTree` | Scans workspace resources. | Recursively enumerates source files in the project workspace root. |
| `Page.getResourceContent` | Retrieves the contents of a workspace file. | Reads text content with strict path-traversal prevention. |

## App Navigation and History

Unlike web applications, standard desktop applications do not have a built-in browser history or a uniform URL-based router. The `Page` domain simulates navigation by interacting with navigation paradigms inside the target Avalonia window.

### Navigation (`Page.navigate`)

When `Page.navigate` is called with a `url` parameter, the CDP server handles it as follows:
1. It retrieves the type of the active session window.
2. It searches for a public method named `Navigate` that accepts a single `string` parameter (e.g., `public void Navigate(string url)`).
3. If found, it invokes this method on the Avalonia UI thread, passing the requested URL.
4. It records the navigation in a session-level history list (`session.NavigationHistory`), updating the current index.
5. It executes any registered lifecycle scripts asynchronously (those added via `addScriptToEvaluateOnNewDocument` or `addScriptToEvaluateOnLoad`).

#### Request Example
```json
{
  "id": 200,
  "method": "Page.navigate",
  "params": {
    "url": "http://127.0.0.1:9222/views/dashboard"
  }
}
```

#### Response Example
```json
{
  "id": 200,
  "result": {
    "frameId": "main-frame-id",
    "loaderId": "main-loader-id"
  }
}
```

### Reloading (`Page.reload`)

When a `Page.reload` command is received, the server forces the active window to fully re-measure, re-arrange, and re-draw its layout. This is accomplished on the Avalonia UI thread:

```csharp
await Dispatcher.UIThread.InvokeAsync(() =>
{
    session.Window.InvalidateMeasure();
    session.Window.InvalidateArrange();
    session.Window.InvalidateVisual();
});
```

Following the visual layout invalidation, the server re-runs all registered scripts on the current document asynchronously.

#### Request Example
```json
{
  "id": 201,
  "method": "Page.reload",
  "params": {}
}
```

#### Response Example
```json
{
  "id": 201,
  "result": {}
}
```

### Navigation History Management

The session maintains a navigation history stack that can be inspected and navigated using:
* `Page.getNavigationHistory`: Returns the current index and list of navigation entries.
* `Page.navigateToHistoryEntry`: Navigates to a specific history entry ID.
* `Page.resetNavigationHistory`: Resets the history list to contain only the root URL.

## High-DPI Screenshots (`Page.captureScreenshot`)

The `Page.captureScreenshot` method takes a screenshot of the active window and returns it as a base64-encoded PNG image. The screenshot is generated by `CaptureScreenshotAsync`.

### High-DPI Scaling Implementation
To ensure captured screenshots remain crisp and match high-resolution layouts (e.g., Retina displays or high-DPI scaling configurations), the CDP server reads the window's current DPI scaling factor (`Window.RenderScaling`).

The scale factor is applied to both the pixel bounds and the DPI vector:
* **Pixel Width:** `Math.Max(1, (int)(Window.Bounds.Width * scale))`
* **Pixel Height:** `Math.Max(1, (int)(Window.Bounds.Height * scale))`
* **DPI Vector:** `96 * scale` (horizontal and vertical)

The server instantiates a `RenderTargetBitmap` with the calculated pixel size and DPI vector, renders the window visual content into it, and saves the image to a memory stream as a PNG.

```csharp
var scale = window.RenderScaling;
var width = Math.Max(1, (int)(window.Bounds.Width * scale));
var height = Math.Max(1, (int)(window.Bounds.Height * scale));

using var bitmap = new RenderTargetBitmap(new PixelSize(width, height), new Vector(96 * scale, 96 * scale));
bitmap.Render(window);
```

> [!NOTE]
> If rendering fails due to platform limitations, headless setups, or initialization errors, the server gracefully tries to generate a fallback mock image of 100x100 pixels colored in Cornflower Blue using SkiaSharp. If SkiaSharp itself is unavailable, it returns a minimal 1x1 black PNG base64 string.

### Clip Regions and Element Screenshots
The standard Chrome DevTools Protocol supports taking cropped screenshots of a specific region using a `clip` parameter. The `clip` parameter contains:
* `x` (number): X offset in CSS pixels.
* `y` (number): Y offset in CSS pixels.
* `width` (number): Width in CSS pixels.
* `height` (number): Height in CSS pixels.
* `scale` (number): Scale factor (e.g., `1` or `2`).

Automated agents and inspectors use the clip coordinates in combination with `DOM.getBoxModel` to extract individual elements. By querying the element's box model coordinates (`model.content` quad), the agent determines the exact coordinates to send in the `clip` parameter of `Page.captureScreenshot`.

#### Request Example (Full Screenshot)
```json
{
  "id": 202,
  "method": "Page.captureScreenshot",
  "params": {
    "format": "png"
  }
}
```

#### Request Example (Element or Clipped Region Screenshot)
```json
{
  "id": 203,
  "method": "Page.captureScreenshot",
  "params": {
    "format": "png",
    "clip": {
      "x": 100,
      "y": 150,
      "width": 300,
      "height": 80,
      "scale": 1.5
    }
  }
}
```

#### Response Example
```json
{
  "id": 202,
  "result": {
    "data": "iVBORw0KGgoAAAANSUhEUgAA..."
  }
}
```

## Live Screencasting (`Page.startScreencast` / `Page.stopScreencast`)

Live screencasting enables streaming real-time video frames of the active application to a remote inspector. This is the underlying mechanism used by the inspector's preview pane. The implementation is located in `StartScreencast` and `StopScreencast` within the `CdpTargetSession` class.

```
+------------------+                   +--------------------+
|  CdpTargetSession|                   | Inspector / Client |
+------------------+                   +--------------------+
         |                                       |
         | --- Page.screencastFrame (Frame 1) -> |
         |                                       |
         | [Wait for Ack or 500ms Watchdog]      |
         |                                       |
         | <--- Page.screencastFrameAck ---------|
         |                                       |
         | --- Page.screencastFrame (Frame 2) -> |
```

### Screencast Configuration Parameters

The `Page.startScreencast` method supports the following options:
* `format` (string, optional): Frame encoding format (`png`, `jpeg`, or `webp`). Defaults to `png`.
* `quality` (integer, optional): Compression quality from `0` to `100` (applicable to lossy formats like `jpeg` and `webp`).
* `maxWidth` (integer, optional): The maximum width of the screencast frames. Images are resized down proportionally.
* `maxHeight` (integer, optional): The maximum height of the screencast frames.
* `everyNthFrame` (integer, optional): Throttles frame rates by only sending 1 out of every N frames.
* `transferMode` (string, optional): Dictates the frame transport strategy (`standard` or `tiled`).

### Backpressure and Flow Control

To prevent network congestion and CPU resource exhaustion, the screencast system implements a built-in backpressure mechanism. 

1. When a new frame is rendered, the server transmits it via a `Page.screencastFrame` event.
2. The server blocks the screencast loop until it receives a corresponding acknowledgment event from the client: `Page.screencastFrameAck` containing the `sessionId` of the frame.
3. If the client fails to acknowledge the frame, a **500ms watchdog timeout** releases the block to prevent the rendering thread from stalling.

### Delta Compression and Change Detection

Before encoding and sending a frame, the server performs change detection to ensure bandwidth is not wasted on static screens. 
* The raw bytes of the newly rendered bitmap are compared against the raw bytes of the previously sent frame (`_lastSentFrameBytes`).
* If the frames are identical, the frame is skipped, the backpressure semaphore is released immediately, and no network packet is dispatched.

### Transfer Modes: Standard vs. Tiled

The `Page.startScreencast` method supports two transfer modes:

#### 1. Standard Transfer Mode
The entire frame is rendered, optionally resized, encoded in the requested format, and transmitted as a single base64 string under the `data` property of the `Page.screencastFrame` event.

#### 2. Tiled Transfer Mode
For larger displays or high-bandwidth connections, the server supports `tiled` screencasts. 
* The viewport is partitioned into a grid of squares (typically 128x128 or 256x256 pixels).
* A specialized class `TiledScreencastProducer` tracks the content of each tile.
* When a frame is captured, the server compares individual tiles against their historical visual state.
* Only the tiles that have actually changed are compressed and transmitted.
* The event contains a `tiles` array containing coordinates, dimensions, and base64-encoded sub-images of only the modified segments.

#### Request Example (Start Screencast)
```json
{
  "id": 204,
  "method": "Page.startScreencast",
  "params": {
    "format": "jpeg",
    "quality": 80,
    "maxWidth": 800,
    "maxHeight": 600,
    "transferMode": "standard"
  }
}
```

#### Event Example (Screencast Frame from Server)
```json
{
  "method": "Page.screencastFrame",
  "params": {
    "data": "/9j/4AAQSkZJRgABAQAAAQABAAD...",
    "metadata": {
      "deviceWidth": 800,
      "deviceHeight": 600,
      "offsetTop": 0,
      "pageScaleFactor": 1,
      "scrollX": 0,
      "scrollY": 0,
      "timestamp": 1782928372
    },
    "sessionId": 42
  }
}
```

#### Acknowledgment Example (Client to Server)
```json
{
  "id": 205,
  "method": "Page.screencastFrameAck",
  "params": {
    "sessionId": 42
  }
}
```

#### Stop Screencast Request
```json
{
  "id": 206,
  "method": "Page.stopScreencast",
  "params": {}
}
```

## Workspace Resources and Content

The Page domain includes diagnostic methods to inspect the directory and source code files of the running project. This allows developers to browse XAML, C#, and configuration files from the DevTools panel.

### `Page.getResourceTree`
Returns a hierarchical resource tree. The server searches upwards from the current directory to locate the project workspace root (by looking for `.sln`, `.slnx`, or `.git` files/directories). Once located, it scans the filesystem recursively (excluding `bin`, `obj`, `.git`, and `node_modules` folders) and catalogs file URLs and mime-types.

### `Page.getResourceContent`
Retrieves the content of any resource URL found in the resource tree. To ensure system security, the server applies path-traversal guards:
1. It validates that the URL begins with the workspace protocol prefix (e.g., `http://127.0.0.1:9222/workspace/`).
2. It constructs the absolute file path and verifies that it resolves to a location within the workspace root.
3. If path traversal is detected (e.g., `..` parameters), access is denied, throwing an exception.

## Lifecycle Scripts

Automated tools often need to inject scripts that run automatically during loading. The Page domain implements these hooks:

* **`addScriptToEvaluateOnNewDocument`**: Registers a script that runs immediately before any page elements are constructed. Returns an identifier.
* **`addScriptToEvaluateOnLoad`**: Registers a script to execute once the page and its visual tree have fully loaded.
* **`removeScriptToEvaluateOnNewDocument`** / **`removeScriptToEvaluateOnLoad`**: Unregisters a previously added script using its identifier.

These scripts consist of C# evaluation strings executing in the target application context, rather than standard browser JavaScript.
