# Browser Inspector (WASM)

The **Browser Inspector** distribution allows running the entire CDP Inspector client within standard modern web browsers via WebAssembly (WASM). Built on top of the .NET WebAssembly workload, this deployment enables cross-platform UI inspection and manual control without requiring users to download native desktop binaries. 

However, running within a browser container introduces strict sandboxing limits. This article covers compiling, publishing, hosting, security limits, and target connection workarounds.

---

## 1. Overview of the Browser Inspector Project

The browser-specific project and layout controls are managed in these directories:
*   **WASM Project**: `CdpInspectorApp.Browser.csproj` configures the WASM build, importing static assets and bundling the compiled Avalonia application.
*   **Connection Control ViewModel**: `ConnectionViewModel.cs` contains the connection lifecycle state machine and toggles direct connection parameters.
*   **Connection View Control**: `ConnectionToolbar.axaml` provides the user-facing inputs to target loopback ports or paste direct WebSocket endpoints.

---

## 2. Compilation, Publishing, and Hosting

To compile the CDP Inspector for browser execution, the machine must have the .NET WebAssembly workloads installed.

### Step 1: Install WASM Build Tools
From the terminal, run the workload installation command:
```bash
dotnet workload install wasm-tools
```
This installs the AOT (Ahead-of-Time) compiler, Emscripten tools, and runtime configurations needed to compile C# assemblies into WebAssembly modules.

### Step 2: Publish the Browser Project
Publish the project in `Release` configuration targeting the output directory:
```bash
dotnet publish samples/CdpInspectorApp.Browser/CdpInspectorApp.Browser.csproj -c Release -o publish
```
This command compiles the C# codebase and outputs the WebAssembly binary assets, HTML index wrappers, and static resources into the `publish/wwwroot` folder.

### Step 3: Serve the Application Locally
Since WASM files must be loaded via HTTP/HTTPS protocols (loading via file paths e.g., `file:///` is blocked by browser security), you must run a local web server. You can run one using `npx`:
```bash
npx http-server publish/wwwroot -p 8080 -c-1
```
The arguments configure:
*   `-p 8080`: Serves the application on port `8080`.
*   `-c-1`: Disables browser caching to make sure you see code changes immediately during debugging.

Once started, navigate to `http://127.0.0.1:8080` in your web browser to load the inspector client.

## 3. Outbound-Only Mode Constraints

A browser-hosted WebAssembly application operates in a sandboxed execution context known as **Outbound-Only Mode**. This sandboxing has several consequences:

1.  **No Server Port Listening**: The browser client cannot open incoming ports or act as a local CDP server. It can only initiate outbound connections.
2.  **No Local Process Spawning**: The inspector cannot launch external target processes. Features inside `AppLauncherService.cs` that spawn target applications (like `CdpSampleApp`) are disabled or throw exceptions in the browser.
3.  **No Direct File System Access**: The browser cannot write test reports directly to the local project path. Replay reports generated by Test Studio must be downloaded through browser file download prompts rather than saved to disk.
4.  **Loopback Scanning Restrictions**: The browser prevents direct, raw TCP connections. All communications must run over standard HTTP(S) or WebSocket protocols.

### Browser vs. Desktop Feature Matrix

| Feature | Browser WASM Client | Desktop Native Client |
| :--- | :---: | :---: |
| **Outbound WebSockets** | Yes | Yes |
| **Target Port Scanning** | Restricted (CORS) | Yes (Unrestricted) |
| **Process Launching** | No | Yes |
| **Local File Saving** | No (Download Prompt) | Yes (Direct File I/O) |
| **Screencast Frame Buffer** | WASM Canvas | SkiaSharp Canvas |
| **Port Listening** | No | Yes |

## 4. Browser Security & CORS Restrictions

When trying to discover target applications (e.g., scanning `http://127.0.0.1:9222`), the browser security engine blocks the connection due to Cross-Origin Resource Sharing (CORS) rules.

### The CORS Discovery Block
*   **The Scenario**: The inspector page is loaded from origin `http://127.0.0.1:8080`. The target application (e.g., `CdpSampleApp`) runs its embedded CDP server on port `9222`.
*   **The Action**: When clicking **Scan Targets**, the inspector attempts an HTTP GET request to `http://127.0.0.1:9222/json` to discover active windows.
*   **The Block**: The browser interceptor flags this as a cross-origin request. Because the target application's lightweight CDP server does not return the HTTP response header `Access-Control-Allow-Origin: *` (or matching origin configurations), the browser blocks the request and hides the returned JSON from the WebAssembly client.

### WebSocket Site-Hijacking Mitigation
Additionally, modern browser engines (including Chromium, WebKit, and Gecko) enforce protections against site-hijacking:
*   Standard web pages are blocked from opening direct WebSocket connections to loopback debugging ports (like `127.0.0.1:9222`) if the connection handshake doesn't include specific security tokens.
*   This prevents malicious public web pages from connecting to your local developer port and executing arbitrary commands.

## 5. WebSocket Connection Workaround & Bypassing Blocks

To bypass these browser blocks, the inspector connection toolbar provides a **Direct Connection** mode. This bypasses origin discovery scanning by targeting WebSocket debugger URLs directly.

### Target Discovery Payloads
When a direct navigation occurs, the target's `/json` or `/json/list` discovery endpoint returns a payload listing all active debugging pages:

```json
[
  {
    "description": "Main Window",
    "id": "e9be9db9-9c59-4d64-a690-349f2b842cd8",
    "title": "CdpSampleApp - Home",
    "type": "page",
    "webSocketDebuggerUrl": "ws://127.0.0.1:9222/devtools/page/e9be9db9-9c59-4d64-a690-349f2b842cd8"
  }
]
```

### The Connection Steps

To connect the browser-hosted inspector to a local target:

```text
+-------------------+                                  +-------------------+
| Browser Inspector |                                  |  Target App Port  |
| (Port 8080 Tab)   |                                  |  (Port 9222 Tab)  |
+-------------------+                                  +-------------------+
          |                                                      |
          |-- [1] Navigate directly to discovery url ------------>|
          |   (User opens http://127.0.0.1:9222/json in new tab) |
          |                                                      |
          |<-- [2] Browser renders discovery JSON page ----------|
          |   (Bypasses CORS because it is a direct navigation)  |
          |                                                      |
          |-- [3] User copies webSocketDebuggerUrl or targetId -->|
          |                                                      |
          |-- [4] Returns to Inspector tab --------------------->|
          |                                                      |
          |-- [5] Pastes URL into connection text box ---------->|
          |   - Selects "Direct Connection" option               |
          |   - Clicks "Connect"                                 |
          |                                                      |
          |-- [6] Initializes WebSocket Handshake -------------->|
```

1.  **Direct Navigation**: Open a new browser tab and navigate directly to the target application's discovery endpoint: `http://127.0.0.1:9222/json` (or `http://127.0.0.1:9223/json` if connecting to another inspector). Because this is a direct browser navigation (and not an asynchronous cross-origin fetch), the browser allows it and renders the JSON output.
2.  **Copy the Endpoint**: From the returned JSON array, identify the target window and copy either the `id` value (e.g. `e9be9db9-9c59-4d64-a690-349f2b842cd8`) or the complete connection string:
    ```text
    ws://127.0.0.1:9222/devtools/page/e9be9db9-9c59-4d64-a690-349f2b842cd8
    ```
3.  **Direct Connection Setup**: Return to the inspector tab at `http://127.0.0.1:8080`.
    *   Paste the copied value into the **Host / WebSocket URL** input textbox.
    *   Toggle the connection mode to **Direct Connection**.
    *   Click **Connect**.
4.  **Handshake Execution**: The WebAssembly client uses standard browser WebSocket APIs to connect directly to the target. By targeting the specific WebSocket URL directly, the connection avoids the CORS block and establishes a JSON-RPC session.
5.  **Enable Domains**: Once connected, the client initializes the debugging session by enabling the required domains (`DOM`, `Input`, `Runtime`, `Page`).

## 6. Content Security Policy (CSP) Considerations

In some corporate environments or production hosting servers, custom Content Security Policy (CSP) headers are defined. Developers should ensure the following directive values are configured on the hosting server:
*   `connect-src 'self' ws://127.0.0.1:* http://127.0.0.1:*`: Allows the WebAssembly client to open WebSocket connections to loopback target ports.
*   `script-src 'self' 'unsafe-eval' 'unsafe-inline'`: Required to load the .NET WebAssembly runtime libraries correctly.

If connections fail immediately, check the browser's developer console (F12) network tab to identify whether CSP or local antivirus filters are intercepting the loopback WebSocket requests.
